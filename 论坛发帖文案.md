【线程优化++】AppOpt 重构版

之前一直在用 Suto 大佬的 AppOpt 和 Aloazny 的二改版，用久了发现有些地方可以再打磨一下，就自己动手改了改。

【改了啥】

主要是在 Aloazny 二改版的基础上重构了底层，没什么大创新，就是让跑起来更稳一点：

· 修了内存泄漏，不会越跑越吃内存了（虽然感觉也没必要哈哈哈）
· 加了前后台判断，切后台自动丢小核，省电少发烫（我自己测没问题）
· 轮询改成了自适应的，线程没变化就拉长间隔，有变化立刻恢复，不至于空转
· 同进程线程数没变就跳过 I/O，不用每次都读 /proc，实测 ~80% 能跳过
· 架构适配从硬编码改成占位符，删了六七十行 sed，以后新架构不用改脚本

对比一下：

          AppOpt原版  Aloazny版  本模块
体积        ~2.4MB     ~45KB     ~45KB
内存泄漏      有         有       修了
前后台感知    无         无        有
自适应轮询    无         无        有
智能I/O复用   无       有缓存    80%跳过
架构适配    硬编码      硬编码    占位符
通配符/Flags  无         有        有

规则方面就是在 Aloazny 那 500+ 应用的基础上继续维护，该有的都有。

【怎么装】

跟普通 Magisk 模块一样，下载 zip 刷入重启就行。KSU、APatch 也能用。

【Flags 开关】

在 /data/adb/modules/Thread-Optimization/Flags/ 下面创建/删除空文件，改完要重新刷一次模块：

  modtify_config      自动适配 CPU 核心配置（默认开）
  update_config       强制用模块规则覆盖你的修改（默认开）
  keep_custom_rule    增量更新，保留你自己加的规则
  delete_game_config  删除所有游戏配置
  dexota_modtify      优化 dex2oat

自己加规则编辑 applist.prop，一行一个：
  com.xxx.xxx{RenderThread}=6-7
  com.xxx.xxx{Thread-*}=4-6

【缺点和鸡肋的地方】

丑话说前头，这玩意有硬伤，不是什么万能药：

1. 同名线程没法区分
当一个进程有多个同名线程时，规则只能写一行，全绑一个核上反而卡。
比如 com.tencent.tmgp.dfm 里有 4 个 GameThread，写了 {GameThread}=7 四个全挤 CPU7，直接完蛋。

2. 配置太死板
一个配置文件几千行，维护起来想死。游戏更新改个线程名，规则直接废了，嘿嘿。
还有着色器线程写 0-5，好家伙直接 6 核全占，跟没优化一个样。

3. 绑单核容易翻车
比如 UnityMain=7，CPU7 在处理中断 / 内核程序 / 其他固定绑核程序时，UnityMain 等 8~16ms 得不到响应就会卡。CPU7 上其他程序频繁活跃，上下文切换飙高，也卡。

4. 最终还是 powerhal / EAS 说了算
就算你写 UnityMain=4-7，EAS 抽风了照样来回切核心跟你写的较劲，写了个寂寞。真正决定线程跑哪个核的，永远是系统调度器。

所以这模块到底有啥用？日用应用减减压，后台少跑大核省省电还行。想靠它游戏质变，期望别太高。
